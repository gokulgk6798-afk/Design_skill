# Milestone 8 Design Patterns and Anti-Patterns Source

This document captures the approved rule for design patterns and anti-patterns provided for Milestone 8.

## Approved Interpretation

Design patterns are derived from the approved Design Kit.

Any approved component, layout, token, content rule, accessibility rule, or interaction behavior documented in the Design Kit is treated as an approved design pattern.

Any design, layout, copy, component usage, token usage, or interaction behavior that conflicts with the approved Design Kit standards is treated as an anti-pattern.

## Pattern Definition

An approved pattern is any documented standard that uses the Design Kit as the source of truth, including:

- approved components
- approved design tokens
- approved color roles
- approved typography
- approved grid, spacing, radius, and elevation rules
- approved UX writing rules
- approved accessibility standards
- approved documentation standards
- approved presentation standards

## Anti-Pattern Definition

An anti-pattern is any implementation or generated artifact that:

- bypasses an approved Design Kit standard
- creates custom UI when an approved component exists
- introduces unapproved colors, spacing, typography, radius, shadows, or layout behavior
- uses inconsistent terminology or UX writing
- relies on color alone to communicate meaning
- creates undocumented variants, states, or interactions without approval
- conflicts with accessibility, documentation, brand, or component standards

## Examples

| Approved Pattern | Anti-Pattern |
| --- | --- |
| Use documented Button component for actions. | Create a custom clickable card styled like a button. |
| Use approved alert anatomy. | Use colored text only with no alert structure. |
| Use approved semantic color roles. | Use random red, green, or yellow values. |
| Use approved grid and spacing. | Manually position elements with arbitrary spacing. |
| Use approved UX terminology. | Mix Employee, Staff, Associate, and User for the same concept. |
| Use approved empty-state format. | Show only `No Data`. |
| Use documented form labels. | Use placeholder-only input labels. |
| Use accessibility color rules. | Communicate status by color alone. |
